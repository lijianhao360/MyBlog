💡 补充说明（方便理解）：

  • char 是定长存储，适合长度固定的字段（如手机号、身份证号），性能比 varchar 略高；

  • varchar 是变长存储，适合长度不确定的文本，会额外占用1-2字节记录长度；

  • blob 系列用于存储二进制数据（如图片、文件），text 系列用于存储纯文本内容。

💡 补充小提示：
datetime 和 timestamp 都能存储日期+时间，核心区别在于：

  • datetime 存储的是「字面时间」，不受时区影响；
  • timestamp 存储的是「Unix时间戳」，会根据数据库时区自动转换，且存在 2038 年时间戳溢出问题。

  1. 概念：约束是作用于表中字段上的规则，用于限制存储在表中的数据。

  • SQL：select * from user where name = 'Arm'; 查询流程（InnoDB聚簇+二级索引）
    1、检索二级索引（name字段为二级索引）
  1. 从name的二级索引B+树根节点开始向下查找，通过B+树有序检索，找到叶子节点 Arm；
  2. 二级索引叶子只存储【索引列name + 主键值】，查询得到Arm对应的主键ID = 10，没有表中其他字段数据。

2、回表查询（重点：select * 需要全字段，触发回表）
因为SQL使用select *，需要整条行所有字段，二级索引缺少其他列数据，需要拿着主键10去聚集索引查完整数据：

  1. 用主键10去聚集索引B+树检索；
  2. 聚集索引叶子节点保存完整整行数据，根据主键10定位到完整行记录；

3、返回结果
取出整行所有字段数据，作为最终查询结果返回。

名词总结
回表：二级索引查出来主键，再凭借主键去聚集索引读取完整数据的过程。

优化：如果改成select 主键,name from user where name='Arm'，只需要二级索引的数据，不需要回表（覆盖索引）。

  1. 失效根本原因
    当where条件中，字符串类型字段（如varchar/char）与数值直接比较时，MySQL会触发隐式类型转换：
  • 自动将索引列的字符串值转换为数值，再和条件中的数值进行比较。
  • 而索引是基于原始字符串的有序结构构建的，转换后的值无法匹配索引的有序性，导致优化器无法利用索引，最终触发全表扫描。

    MySQL的比较规则中，当字符串类型字段和数值比较时，会执行类似字段 + 0的隐式转换（将字符串转成数值）。而索引列一旦被函数/运算处理，就会失去有序性，优化器无法通过索引快速定位数据。

2.9 覆盖索引和回表查询

  • 覆盖索引
    尽量使用覆盖索引（查询使用了索引，并且需要返回的列，在该索引中已经全部能够找到），减少select *。
  1. 核心定义
    覆盖索引：查询使用的索引中，已经包含了查询所需的所有列（包括 SELECT 列和 WHERE 条件列），无需再回表查询聚簇索引数据。
    一句话总结：索引里有你要的所有数据，直接读索引就能拿到结果，不用再去查原表。
  1. 核心作用
  • 避免回表操作（减少随机IO，大幅提升查询性能）
  • 减少磁盘IO：索引通常比聚簇索引小，能缓存更多数据
  • 降低锁竞争：不需要访问原表，减少行锁冲突概率
  1. 关键规则：尽量减少 SELECT *
    SELECT * 会查询所有列，而二级索引通常只包含索引列和主键，无法覆盖所有列，必然触发回表操作，失去覆盖索引的优势。
  1. 示例讲解（以联合索引 idx_pas(profession, age, status) 为例）
    前提：假设表 tb_user 上有联合索引 idx_pas(profession, age, status)，二级索引的叶子节点会存储 profession, age, status + 主键 id。

2.10 前缀索引

当字段类型为字符串（varchar，text等）时，有时候需要索引很长的字符串，这会让索引变得很大，查询时，浪费大量的磁盘IO， 影响查询效率。此时可以只将字符串的一部分前缀，建立索引，这样可以大大节约索引空间，从而提高索引效率。

语法

前缀长度

  • 可以根据索引的选择性来决定，而选择性是指不重复的索引值（基数）和数据表的记录总数的比值，索引选择性越高则查询效率越高，唯一索引的选择性是1，这是最好的索引选择性，性能也是最好的。

  • QL语句作用解析（前缀索引场景）

一、核心概念：索引选择性
索引选择性（区分度）= 不重复的索引值数量 / 表总记录数

  • 取值范围：0 ~ 1，越接近1，索引效率越高
  • 唯一索引的选择性为1，是最优状态
  • 选择性过低的索引，会导致优化器放弃使用索引，退化为全表扫描

  1. 第一句：计算原字段的基准选择性

  • 计算结果：email字段本身的索引选择性，作为前缀索引的“基准参考值”
  1. 第二句：计算指定长度前缀的选择性

  • count(distinct substring(email,1,5))：统计前5个字符的不重复值数量
  • 计算结果：长度为5的前缀索引的选择性，用于和基准值对比，评估区分度损失

2.12 索引设计原则

  1. 针对于数据量较大，且查询比较频繁的表建立索引。
  2. 针对于常作为查询条件（where）、排序（order by）、分组（group by）操作的字段建立索引。
  3. 尽量选择区分度高的列作为索引，尽量建立唯一索引，区分度越高，使用索引的效率越高。
  4. 如果是字符串类型的字段，字段的长度较长，可以针对于字段的特点，建立前缀索引。
  5. 尽量使用联合索引，减少单列索引，查询时，联合索引很多时候可以覆盖索引，节省存储空间，避免回表，提高查询效率。
  6. 要控制索引的数量，索引并不是多多益善，索引越多，维护索引结构的代价也就越大，会影响增删改的效率。
  7. 如果索引列不能存储NULL值，请在创建表时使用NOT NULL约束它。当优化器知道每列是否包含NULL值时，它可以更好地确定哪个索引最有效地用于查询。

（1）联合索引（最左前缀原则，高频）

  • 核心规则：查询条件必须包含联合索引最左侧的列，索引才会生效；
  • 例子：联合索引(a,b,c)，where a=1 → 索引生效；where b=2 → 索引失效；where a=1 and b=2 → 索引生效；
  • 面试常问：联合索引的顺序怎么设计？
    遵循**“等值在前，范围在后”**原则，比如where a=1 and b>2，建索引(a,b)比(b,a)更优。

（2）索引失效（高频面试题，常见场景）

  1. 对索引列进行了运算、函数操作，比如where year(create_time) = 2024 → 索引失效；
  2. 索引列发生隐式类型转换，比如字段是varchar，查询用where phone=17799990010（数字），索引失效；
  3. 使用OR连接条件，其中一个条件的列无索引；
  4. LIKE查询以%开头，比如where name like '%张' → 索引失效；where name like '张%' 索引生效；
  5. 索引列使用!=、<>、not in，优化器可能放弃索引。

（3）覆盖索引

  • 定义：索引中包含了查询所需的所有列，无需回表；
  • 面试常问：覆盖索引的好处？
    1. 避免回表，减少磁盘IO，大幅提升查询性能；
    2. 减少锁竞争，不需要访问原表数据；
  • 例子：select id, name from user where name='张三'，如果索引是idx_name_id(name,id)，就可以覆盖，Extra字段会显示Using index。

（4）前缀索引

  • 定义：对字符串的前N个字符建立索引，解决长字符串索引过大的问题；
  • 面试常问：前缀索引的缺点？
    1. 无法用于ORDER BY、GROUP BY；
    2. 无法作为覆盖索引，因为只存储了前缀，无法获取完整数据。

（5）单列vs联合索引
多条件查询优先使用联合索引，避免多个单列索引的优化器选择问题，同时减少索引数量。

    1. 索引设计原则
      💡 面试必背7条原则
  1. 为数据量大、查询频繁的表建立索引；
  2. 为WHERE、ORDER BY、GROUP BY、JOIN条件的列建立索引；
  3. 优先为区分度高的列建立索引（区分度=不重复值数量/总记录数，越高越好）；
  4. 长字符串使用前缀索引，控制索引体积；
  5. 优先使用联合索引，减少单列索引；
  6. 控制索引数量，索引会降低增删改效率，避免过度建索引；
  7. 索引列尽量设置为NOT NULL，优化器可以更好地利用索引。

3.3.order by优化

  • order by优化准则
  • 根据排序字段建立合适的索引，多字段排序时，也遵循最左前缀法则。
  • 尽量使用覆盖索引。
  • 多字段排序, 一个升序一个降序，此时需要注意联合索引在创建时的规则（ASC/DESC）。
  • 如果不可避免的出现filesort，大数据量排序时，可以适当增大排序缓冲区大小 sort_buffer_size(默认256k)。
  1. 最左前缀法则（重点！没有最右，只有最左）
  • 定义：使用联合索引时，必须从索引的第一个字段开始，连续使用，跳过左边的字段直接用右边的，索引直接失效。
  • 排序场景举例：
    建了联合索引 (age, phone)
    有效：order by age, phone（从左到右连续用）
    失效：order by phone（跳过age，直接用phone）
  • 一句话记：联合索引/排序，必须从左往右用，不能跳。
  1. 覆盖索引
  • 定义：你查询的所有字段，全都在索引里面，不需要回表查原数据。
  • 排序优势：
    索引本身是有序的，直接拿索引数据就能返回结果，不用回表、不用额外排序，速度极快。
  • 举例：
    索引 (age, phone) 包含了id、age、phone
    select id,age,phone from user order by age,phone → 完美覆盖索引。
  1. Using filesort（文件排序）
  • 定义：没有用到索引排序，MySQL只能把数据读到内存/磁盘里，重新手动排序。
  • 特点：效率极低，是排序优化要极力避免的情况。
  • 一句话记：出现 Using filesort = 排序慢，要优化。
  1. sort_buffer_size
  • 定义：MySQL给filesort排序专用的内存缓冲区。
  • 默认大小：256KB
  • 作用：缓冲区越大，能在内存里排序的数据越多，越少用到磁盘排序，速度越快。
  • 适用场景：实在避免不了filesort时，调大它提升排序速度。
  1. 联合索引排序规则（ASC/DESC）
  • 定义：多字段排序时，如果一个升序、一个降序，创建联合索引时，必须严格对应排序方向，否则索引失效。
  • 举例：
    排序：order by age asc, phone desc
    建索引必须写：index (age asc, phone desc)

3.5 limit 优化

一、问题场景：大偏移量分页查询
当执行类似 limit 2000000, 10 的查询时，会出现严重的性能问题：

  • 执行逻辑：MySQL需要先排序并读取前 2000010 条记录，再丢弃前 2000000 条，仅返回最后10条。
  • 性能瓶颈：大量数据被读取后丢弃，排序和IO的代价极高。
  • 示例：select id from tb_sku order by id limit 9000000,10; 执行耗时约11.47秒。

二、优化思路
分页查询时，通过覆盖索引 + 子查询的形式，可以大幅提升性能：

  1. 利用覆盖索引，让子查询仅读取索引列（如id），快速定位到目标数据的主键；
  2. 再通过主键回表查询完整数据，避免全表排序和大量数据读取。

四、核心原理

  • 覆盖索引：子查询 select id from tb_sku order by id limit 2000000,10 仅访问主键索引，无需回表，IO开销极低；
  • 主键关联：外层查询通过子查询返回的id，直接回表读取10条完整数据，避免了读取和丢弃前面200万条数据的开销。

3.6 count优化
MySQL count优化笔记

一、不同存储引擎的count(*)差异
测试SQL

  1. MyISAM引擎
  • 特性：会把表的总行数直接存储在磁盘上
  • 执行count(*)时，直接读取预存的行数返回，效率极高
  1. InnoDB引擎
  • 特性：事务性引擎，存在MVCC机制，无法预存总行数（不同事务可见的行数不同）
  • 执行count(*)时，需要逐行读取数据并累积计数，效率较低

二、count函数的原理与常见用法
核心原理
count()是聚合函数，对查询结果集逐行判断：

  • 若count的参数不为NULL，累计值+1
  • 若参数为NULL，则不累加
  • 最终返回累计的行数

常见用法
count(*)、count(主键)、count(字段)、count(1)

三、各用法的执行细节与效率对比

  1. count(主键)
  • 执行逻辑：InnoDB遍历全表，取出每行的主键ID（主键非NULL），服务层直接按行累加
  • 开销：需要读取主键字段值，存在一定IO开销
  1. count(字段)
  • 无NOT NULL约束：遍历全表取出字段值，服务层判断字段非NULL后才累加
  • 有NOT NULL约束：遍历全表取出字段值，直接按行累加
  • 开销：需要读取字段值，且存在NULL判断逻辑，效率最低
  1. count(1)
  • 执行逻辑：InnoDB遍历全表，不读取任何字段值，服务层直接为每行填充数字1并累加
  • 开销：无字段读取操作，仅遍历行
  1. count(*)
  • 执行逻辑：InnoDB专门对count(*)做了优化，不读取任何字段值，直接按行累加计数
  • 开销：和count(1)几乎一致，是InnoDB中最优的count用法

效率排序
count(字段) < count(主键id) < count(1) ≈ count(*)
最佳实践：优先使用count(*)

四、优化思路
对于InnoDB引擎的大表count(*)查询，可采用应用层自己计数的方式：

  • 在业务表中维护一个计数器字段，每次增删改时同步更新
  • 查询总数时直接读取计数器字段，避免全表扫描
  • 适用于查询频率高、数据量大的场景

4.2 检查选项(cascaded和local)

当使用WITH CHECK OPTION子句创建视图时，MySQL会通过视图检查正在更改的每个行（例如插入、更新、删除），以使其符合视图的定义。MySQL允许基于另一个视图创建视图，它还会检查依赖视图中的规则以保持一致性。为了确定检查的范围，MySQL提供了两个选项：CASCADED 和 LOCAL，默认值为 CASCADED。

  • cascaded
  • local
  • 一、核心基础概念
  1. WITH CHECK OPTION
    视图插入/更新时，校验数据是否符合视图的 WHERE 条件，不满足则拒绝操作，保证数据一致性。
  2. 无检查选项
    不做任何校验，数据直接写入基表；不满足条件的数据 → 视图不可见，基表存在。
  • 二、两大检查选项规则
    选项 核心规则 严格程度
    LOCAL 仅检查当前视图条件，不检查父视图 中等
    CASCADED 检查当前视图+所有父视图，默认值，最严格 最高
  • 三、四大场景 面试必背总结
    场景组合 检查规则 结果
    父 LOCAL + 子 无检查 子无检查 → 不校验任何条件 全部插入成功
    父 无检查 + 子 LOCAL 仅校验子视图，无视父视图 满足子视图即可插入
    父 CASCADED + 子 无检查 子无检查 → 不校验任何条件 全部插入成功
    父 无检查 + 子 CASCADED 强制校验子+父所有条件 全部满足才允许插入
  • 四、面试高频易错点（死记硬背）
  1. 检查选项不继承
    父视图加了 LOCAL/CASCADED，子视图无检查选项 → 完全不校验，直接插入。
  2. CASCADED 强制递归
    子视图加 CASCADED，无论父视图有没有检查选项，都会检查所有父视图条件。
  3. LOCAL 完全隔离
    子视图加 LOCAL，只看自己，父视图的条件不影响插入。

4.3 视图更新及作用

  • 1.视图更新
    要使视图可更新，视图中的行与基础表中的行之间必须存在一对一的关系。如果视图包含以下任何一项，则该视图不可更新：
  1. 聚合函数或窗口函数（SUM()、MIN()、MAX()、COUNT()等）
  2. DISTINCT
  3. GROUP BY
  4. HAVING
  5. UNION 或者 UNION ALL
  • 2.MySQL 视图三大作用

视图是数据库的“虚拟表”，不存储真实数据，核心作用可概括为简化操作、权限隔离、解耦结构，是面试高频考点。

    1. 简化：降低操作与理解成本
      【核心】封装常用/复杂查询，减少重复SQL编写，简化业务操作。

  • 把高频使用的复杂查询（如多表联查、多条件过滤）定义为视图，用户直接查询视图即可，无需每次编写完整SQL。

  • 屏蔽底层多表关联的复杂性，让用户像操作单表一样使用数据，降低数据理解门槛。
    【面试场景】订单系统中，将“订单表+用户表+商品表”的联查封装为视图，业务方直接调用视图即可，无需重复编写联查逻辑。

    1. 安全：实现精细化数据权限控制
      【核心】弥补数据库原生授权的不足，实现行级/列级数据隔离。

  • 数据库原生GRANT授权仅支持到表级别，无法限制用户访问特定行/列的数据。

  • 通过视图可过滤敏感列（如隐藏手机号、身份证号）、限制数据范围（如员工只能查看自己的考勤数据），用户仅能访问视图可见的数据，避免敏感数据泄露。
    【面试场景】电商后台中，客服人员只能查看用户订单的基本信息，无法看到用户的手机号、地址等隐私数据，通过视图即可实现。

    1. 数据独立：屏蔽底层表结构变化
      【核心】解耦业务查询与底层表结构，降低结构变更对业务的影响。

  • 当底层真实表发生结构变更（如字段改名、表拆分、字段调整），只需修改视图的定义，上层业务的查询逻辑无需修改，即可保持正常运行。

    【面试场景】用户表拆分后，通过视图整合新旧表数据，上层业务无需修改SQL，就能正常查询用户信息。

5.锁

5.1 锁概述

  • 1.介绍
    锁是计算机协调多个进程或线程并发访问某一资源的机制。在数据库中，除传统的计算资源（CPU、RAM、I/O）的争用以外，数据也是一种供许多用户共享的资源。如何保证数据并发访问的一致性、有效性是所有数据库必须解决的一个问题，锁冲突也是影响数据库并发访问性能的一个重要因素。从这个角度来说，锁对数据库而言显得尤其重要，也更加复杂。

  • 2.分类
    MySQL中的锁，按照锁的粒度分，分为以下三类：

  1. 全局锁：锁定数据库中的所有表。
  2. 表级锁：每次操作锁住整张表。
  3. 行级锁：每次操作锁住对应的行数据。

5.2 全局锁

  • 1.介绍
    全局锁就是对整个数据库实例加锁，加锁后整个实例就处于只读状态(DQL)，后续的DML的写语句，DDL语句，已经更新操作的事务提交语句都将被阻塞。
    其典型的使用场景是做全库的逻辑备份，对所有的表进行锁定，从而获取一致性视图，保证数据的完整性。

    数据库中加全局锁，是一个比较重的操作，存在以下问题；【解释：全局锁会锁定库内全部表，阻塞所有写操作，对业务影响极大】

  1. 如果在主库上备份，那么在备份期间都不能执行更新，业务基本上就得停摆。【解释：主库负责线上业务写入，写操作被阻塞会直接阻断正常业务更新】
  2. 如果在从库上备份，那么在备份期间从库不能执行主库同步过来的二进制日志（binlog），会导致主从延迟。【解释：从库无法同步主库数据，主从库数据产生时间差，一致性下降】

在InnoDB引擎中，我们可以在备份时加上参数 --single-transaction 参数来完成不加锁的一致性数据备份。【解释：借助InnoDB的MVCC快照机制，备份全程不加锁，不影响业务读写】

四、面试重点补充

  1. 适用存储引擎
    MyISAM引擎默认使用表锁；InnoDB也支持手动lock tables表锁，但生产环境极少使用，InnoDB优先使用粒度更小的行锁提升并发。
  2. 性能短板
    锁定粒度为整张表，读写互斥，锁冲突概率极高，数据库并发性能差，只适用于低并发场景。
  3. 与InnoDB行锁核心区别
    lock tables表锁属于会话级锁，事务提交不会自动释放；InnoDB行锁是事务级锁，事务提交/回滚自动释放。
  4. 和MDL元数据锁区分
    lock tables是开发者手动添加的显式表锁；MDL（元数据锁）是访问表时MySQL自动添加的隐式锁，无需手动操作。
    1. 元数据锁（meta data lock，MDL）
      MDL加锁过程是系统自动控制，无需显式使用，在访问一张表的时候会自动加上。MDL锁主要作用是维护表元数据的数据一致性，在表上有活动事务的时候，不可以对元数据进行写入操作。为了避免DML与DDL冲突，保证读写的正确性。

在MySQL5.5中引入了MDL，当对一张表进行增删改查的时候，加MDL读锁(共享)；当对表结构进行变更操作的时候，加MDL写锁(排他)。

5.4 行级锁

  • 1.介绍
    行级锁，每次操作锁住对应的行数据。锁定粒度最小，发生锁冲突的概率最低，并发度最高。应用在InnoDB存储引擎中。

InnoDB的数据是基于索引组织的，行锁是通过对索引上的索引项加锁来实现的，而不是对记录加的锁。对于行级锁，主要分为以下三类：

  1. 行锁（Record Lock）：锁定单个行记录的锁，防止其他事务对此行进行update和delete。在RC、RR隔离级别下都支持。
  2. 间隙锁（Gap Lock）：锁定索引记录间隙（不含该记录），确保索引记录间隙不变，防止其他事务在这个间隙进行insert，产生幻读。在RR隔离级别下都支持。
  3. 临键锁（Next-Key Lock）：行锁和间隙锁组合，同时锁住数据，并锁住数据前面的间隙Gap。在RR隔离级别下支持。
  • 2.行锁

InnoDB实现了以下两种类型的行锁：

  1. 共享锁（S）：允许一个事务去读一行，阻止其他事务获得相同数据集的排他锁。
  2. 排他锁（X）：允许获取排他锁的事务更新数据，阻止其他事务获得相同数据集的共享锁和排他锁。
  • 一、锁兼容关系（面试必背矩阵）
    当前持有锁 申请共享锁(S) 申请排他锁(X)
    共享锁(S) 兼容（多事务可读） 互斥（阻塞）
    排他锁(X) 互斥（阻塞） 互斥（阻塞）
  • 文字解释
  1. 共享锁S（读锁）
    多个事务可以同时加S锁并行读同一行；但只要该行存在S锁，其他事务不能加X写锁。
  2. 排他锁X（写锁）
    只有持有X锁的事务能读写该行；其他事务不管申请S读锁还是X写锁，全部阻塞。

  1. 行锁生效前提
    条件查询字段必须走索引；如果没命中索引，InnoDB会退化为表锁，整张表被锁住，并发大幅下降（高频面试坑）。

    行锁是事务级锁，只有执行commit/rollback结束事务后才释放，单条SQL执行完不会释放。
  3. 普通select不加锁
    普通select * from tb_user快照读（MVCC）不加任何行锁，无阻塞；只有lock in share mode/for update/DML才会加锁。
  4. 锁粒度优势
    只锁定操作的单行，其他行不受影响，并发度远高于表锁，是InnoDB高性能的核心。

  • InnoDB行锁核心规则（面试笔记+重点详解）
    默认情况下，InnoDB在 REPEATABLE READ事务隔离级别运行，InnoDB使用 next-key 锁进行搜索和索引扫描，以防止幻读。

    重点解释：

    1. MySQL InnoDB默认隔离级别为RR（可重复读）；
    2. RR隔离下范围查询会加临键锁（Next-Key Lock = 行锁Record + 间隙锁Gap），锁住一段索引区间，阻止其他事务在区间内插入新数据，从机制上解决幻读；
    3. RC（读已提交）隔离没有间隙锁，仅存在行锁，无法规避幻读。

  1. 针对唯一索引进行检索时，对已存在的记录进行等值匹配时，将会自动优化为行锁。

    重点拆解：

    • 唯一索引：主键索引、唯一键索引；
    • 条件：等值查询 where id = 1，且表中真实存在这条记录；
    • 优化逻辑：自动去除间隙锁，只加Record单行锁，仅锁定匹配到的那一行，并发更高；
    • 补充坑点：如果等值查询匹配不存在的数据，不会优化为纯行锁，依旧保留间隙锁；普通二级索引等值查询也不会完全消除间隙锁。

  • RR隔离下临键锁退化规则（面试笔记·带重点详解）
    默认情况下，InnoDB在 REPEATABLE READ事务隔离级别运行，InnoDB使用 next-key 锁进行搜索和索引扫描，以防止幻读。

    基础解释：InnoDB默认RR隔离，标准加锁单元是临键锁Next-Key Lock（行锁Record+间隙锁Gap）；引擎会根据索引类型、查询条件、数据是否存在做锁退化优化，缩小锁定范围提升并发。

    1. 唯一索引等值查询，匹配记录不存在 → 锁退化为纯间隙锁
      重点解释

  • 场景：主键/唯一键等值查询，表内没有这条数据；

  • 逻辑：没有匹配的行记录，无需加Record行锁，仅锁住目标值前后的索引间隙，阻止其他事务插入这条数据，杜绝幻读；

  • 示例：表主键id只有1、3、5，执行update tb set name='xx' where id=2;
    不存在id=2，只锁住(1,3)间隙，不会锁定任何真实行。

    1. 普通索引等值查询，遍历到末尾不满足条件 → Next-Key退化为间隙锁
      重点解释

  • 普通索引无唯一性约束，会向后遍历全部等值数据；当遍历到第一条不匹配的索引项时，末尾的临键锁会去掉行锁，仅保留间隙锁；

  • 区别唯一索引：唯一索引找到一条匹配数据就停止遍历，普通索引会遍历全部相同值。

    1. 唯一索引范围查询：会一直扫描到第一个不满足条件的索引项为止
      重点解释
      范围条件（>、<、>=、<=、between）不会完全退化临键锁，会持续向后扫描索引，直到遇到第一条不符合范围的值，沿途所有索引区间都会加临键锁，锁定范围更大；
      例：where id < 5，主键索引会锁住所有小于5的临键区间，直到索引中第一个≥5的记录。

#- 红字核心注意（面试高频问答）
注意：间隙锁唯一目的是防止其他事务插入间隙。间隙锁可以共存，一个事务采用的间隙锁不会阻止另一个事务在同一间隙上采用间隙锁。

  • 拆解考点
  1. 间隙锁作用单一：只堵INSERT插入操作，不阻塞SELECT、UPDATE、DELETE已有行；
  2. 间隙锁互相兼容：多个事务可以同时持有同一个间隙的Gap锁，互不阻塞；
  3. 互斥关系：间隙锁仅和插入意向锁互斥；行锁、临键锁之间才会发生读写阻塞；
  4. 坑点：仅RR隔离存在间隙锁，RC读已提交隔离级别彻底关闭Gap锁，无法避免幻读。
  • 面试速记总结
  1. 唯一索引等值命中存在数据 → 纯行锁；不存在 → 纯间隙锁；
  2. 普通索引等值遍历结束后，末尾临键锁退化间隙锁；
  3. 范围查询不会完全退化临键锁，锁定一整段索引区间；
  4. 间隙锁可共享，只防插入，不阻塞对已有行的修改。

3.事务原理

事务是一组操作的集合，它是一个不可分割的工作单位，事务会把所有的操作作为一个整体一起向系统提交或撤销操作请求，即这些操作要么同时成功，要么同时失败。

  • 特性（ACID四大特性）

  • 原子性（Atomicity）：事务是不可分割的最小操作单元，要么全部成功，要么全部失败。

  • 一致性（Consistency）：事务完成时，必须使所有的数据都保持一致状态。

  • 隔离性（Isolation）：数据库系统提供的隔离机制，保证事务在不受外部并发操作影响的独立环境下运行。

  • 持久性（Durability）：事务一旦提交或回滚，它对数据库中的数据的改变就是永久的。

  • 面试重点补充解释

  1. 原子性 Atomicity
    依托undo回滚日志实现。事务中途失败时，会通过undo日志回滚所有已执行操作，不会出现部分执行成功的情况。
    举例：转账A扣钱、B加钱，其中一步出错，两条SQL全部撤销。
  2. 一致性 Consistency
    是事务最终目标，由原子性、隔离性、持久性共同保障。事务执行前后，数据完整性约束不会被破坏（主键唯一、外键、字段校验、业务余额平衡等）。
  3. 隔离性 Isolation
    依托锁、MVCC多版本并发控制实现。并发事务之间互相隔离，互不干扰；数据库提供4种隔离级别，用来平衡并发性能与数据一致性。
  4. 持久性 Durability
    依托redo重做日志实现。事务提交后修改写入redo日志，即使服务器宕机，重启后可通过redo日志恢复已提交的数据，修改永久生效。

4.MVCC

  • 1.MVCC-基本概念

  • 一、当前读
    原文：
    读取的是记录的最新版本，读取时还要保证其他并发事务不能修改当前记录，会对读取的记录进行加锁。对于我们日常的操作，如：
    select ... lock in share mode(共享锁)，select ... for update、update、insert、delete(排他锁)都是一种当前读。

  • 重点解释

  1. 核心特征：读取当前数据最新提交版本，主动加行锁，读写互斥，会阻塞其他事务的写操作；
  2. 所有写操作（增删改）、手动锁定读（共享锁/排他锁查询）都属于当前读；
  3. 锁机制：依托行级S/X锁实现并发控制，存在读写阻塞，并发性能低。

  • 二、快照读
    原文：
    简单的select（不加锁）就是快照读，快照读，读取的是记录数据的可见版本，有可能是历史数据，不加锁，是非阻塞读。

  • Read Committed（读已提交）：每次select，都生成一个快照读。

  • Repeatable Read（可重复读，InnoDB默认）：开启事务后第一个select语句才生成快照。

  • Serializable（串行化）：快照读会退化为当前读。

  • 重点解释

  1. 核心特征：普通无锁select，不加任何行锁，读不阻塞写、写不阻塞读，大幅提升并发；读取undo日志里的历史快照版本，不一定是最新数据；
  2. RC隔离：每次查询都会重新生成ReadView，能读到其他事务已提交的新数据；
  3. RR隔离：事务第一条select才创建ReadView，整个事务复用同一个快照，保证可重复读；
  4. 串行化隔离级别取消MVCC快照机制，所有查询都升级为加锁当前读，完全规避并发问题，但性能最差。

  • 三、MVCC
    原文：
    全称 Multi-Version Concurrency Control，多版本并发控制。指维护一个数据的多个版本，使得读写操作没有冲突，快照读为MySQL实现MVCC提供了一个非阻塞读功能。MVCC的具体实现，还需要依赖于数据库记录中的三个隐式字段、undo log日志、readView。

  • 重点拆解

  1. 核心作用：解决读写阻塞问题，读不加锁、写只锁当前行，实现高并发；仅在RC、RR隔离生效；
  2. 三大实现依赖：
    1）行隐藏字段：trx_id（事务ID）、roll_pointer（回滚指针）、row_id（无主键时隐藏主键）；
    2）undo log回滚日志：保存数据修改前的历史版本，形成版本链；
    3）ReadView读视图：判断当前事务能看见哪些数据版本，实现隔离性；
  3. 适用范围：InnoDB引擎专属，MyISAM不支持MVCC。

  • 3.undo log

回滚日志，在insert、update、delete的时候产生的便于数据回滚的日志。
当insert的时候，产生的undo log日志只在回滚时需要，在事务提交后，可被立即删除。
而update、delete的时候，产生的undo log日志不仅在回滚时需要，在快照读时也需要，不会立即被删除。

  • 面试重点注释详解

  1. undo log两大核心作用
    ① 保障事务原子性：事务失败时，通过undo日志回滚所有已执行操作，实现全部失败；
    ② 支撑MVCC快照读：保存数据修改前的历史版本，构建版本链，无锁读取历史数据。

  2. INSERT操作的undo日志生命周期
    插入操作的undo仅用于回滚（失败时删除新增行）；事务提交后没有任何事务会再读取这条插入前的空白版本，因此提交后可直接清理删除，无残留。

  3. UPDATE/DELETE操作的undo日志生命周期
    修改、删除产生的undo保存了数据旧版本，其他并发事务的快照读可能需要读取该历史版本，所以事务提交后不能立刻删除；
    只有当系统确认没有任何事务需要访问该undo版本时，才会通过后台purge线程统一清理回收。

  4. 补充考点
    undo日志存储在单独的undo段，可设置独立undo表空间；undo是逻辑日志，记录的是对应的反向SQL，而非物理数据变更。

    1. ReadView 4条可见性判断规则
  1. trx_id == creator_trx_id → 可访问；说明这条数据是当前事务自己修改的
  2. trx_id < min_trx_id → 可访问；说明修改该数据的事务早已提交
  3. trx_id > max_trx_id → 不可访问；修改事务是在ReadView创建后才开启的
  4. min_trx_id ≤ trx_id ≤ max_trx_id：trx_id不在m_ids活跃事务集合内 → 可访问；代表修改事务已提交
    1. 隔离级别标识：RC（READ COMMITTED 读已提交）
  • 二、分层面试注释详解
    1. ReadView 4个字段含义（必背）
  1. creator_trx_id：创建这个ReadView的当前事务ID（本例为5，事务5）
  2. m_ids：生成ReadView时，所有活跃未提交事务ID集合
  3. min_trx_id：活跃事务中最小的事务ID
  4. max_trx_id：下一个即将分配的事务ID（大于所有已存在事务ID）
    1. RC隔离关键特性（图中核心考点）
      RC隔离级别：每执行一次普通SELECT，就重新生成一份全新ReadView
  • 事务5第一次查询：事务3、4、5都未提交，m_ids={3,4,5}
  • 事务5第二次查询：事务3已经提交，只剩4、5活跃，m_ids={4,5}
  • 效果：每次查询都能看见其他事务刚提交完成的最新数据，这就是“读已提交”

对比RR隔离：事务内第一条SELECT才生成ReadView，全程复用同一份视图，不会重复创建。

    1. 四条可见性规则逐条拆解
  1. 自己改的数据永远可见
    当前行的修改事务ID = 当前查询事务ID，直接读取，不用遍历undo版本链。
  2. 事务ID小于最小活跃ID：已提交，可见
    能进入min_trx_id之前的事务，全部执行完并提交，数据是稳定可读取的。
  3. 事务ID大于max_trx_id：完全不可见
    max_trx_id是未来未分配的事务号，这个修改操作发生在视图创建之后，看不到。
  4. 区间内但不在活跃集合：已提交，可见
    事务ID落在min~max之间，但不在m_ids活跃列表，说明该事务中途已经提交，可以读取它修改的数据。
    1. undo版本链作用注释
      当当前行版本不符合可见规则时，顺着DB_ROLL_PTR指针往undo日志里找旧版本，循环套用4条判断规则，直到找到第一条对当前ReadView可见的历史快照，返回给用户（快照读无锁的核心）。
    1. 面试高频区分：RC vs RR 的ReadView差异
      隔离级别 ReadView生成时机 现象
      RC读已提交 每次SELECT都新建 能读到其他事务刚提交的数据，会出现不可重复读
      RR可重复读 事务内第一条SELECT生成，全程复用 整个事务快照固定，解决不可重复读，配合临键锁解决幻读
    1. 整体流程总结
      快照读时 → 生成ReadView → 拿当前行DB_TRX_ID匹配4条规则 → 不可见则遍历undo版本链，直到找到可见版本返回。

    • mysql.time_zone：时区数据、主从复制相关配置

  • 考点：删除该库会直接导致MySQL无法启动、登录、权限失效。

    1. information_schema（元数据库）

  • 元数据定义：描述数据库本身的数据（库名、表名、字段、索引、约束、存储过程等）

  • 常用场景：代码自动生成SQL、查询表结构、统计库表数量、判断索引是否存在

  • 常用表：

    • tables：所有表信息
    • columns：字段信息
    • key_column_usage：外键、主键索引

  • 特点：只读视图，不能修改，重启不会丢失数据。

    1. performance_schema（性能底层采集库）

  • 底层实时采集MySQL运行时所有行为：锁、SQL语句、IO、事务、线程等待、文件读写等

  • 之前讲锁时查询锁的SQL就是用它：performance_schema.data_locks

  • 适用场景：定位锁等待、慢SQL、IO瓶颈、线程阻塞问题

  • 特点：数据是运行时临时采集，重启MySQL数据清空；性能采集会轻微损耗服务器性能。

    1. sys 简化诊断库

  • 本质：基于performance_schema封装的一套易用视图，简化复杂查询，不用手写多表关联

  • 常用视图：

    • sys.schema_unused_indexes：查询无用索引
    • sys.statement_analysis：统计慢SQL、高频SQL耗时
    • sys.innodb_lock_waits：直观查看锁等待、死锁信息

  • 定位：DBA日常调优首选，大幅简化性能排查SQL编写。

  • 面试速记区分

  1. mysql：管账号权限、全局配置；
  2. information_schema：查表结构、元数据；
  3. performance_schema：底层原始性能采集；
  4. sys：封装好的简化性能诊断视图。

    1. mysqladmin 运维管理工具
  • 基础作用
    无需进入MySQL交互窗口，直接在系统命令行执行数据库运维、监控、账号、服务管理操作，DBA运维专用工具。
  • 常用子命令

  2. shutdown是优雅关闭，会刷写redo/undo日志，比强制kill进程安全；
  3. 适合监控脚本定时探测MySQL服务是否宕机。

    1. mysqlbinlog Binlog二进制日志解析工具
  • 基础作用
    解析二进制binlog日志文件，用于误删数据时间点恢复、主从同步故障排查、SQL操作审计。
  • 面试核心考点
  1. Binlog仅记录DDL/DML写操作，普通SELECT查询不会记录；
  2. 三种日志格式：statement、row、mixed，row格式可精准恢复单行数据；
  3. 支持按时间、日志偏移位置截取日志做精准数据回滚。
  • 常用参数
    --start-datetime、--stop-datetime：按时间截取日志；
    --start-position、--stop-position：按日志偏移位置截取；
    -v/-vv：展示行级详细修改SQL。

  • 基础作用

    1. mysqldump 逻辑备份导出工具（面试最高频）
  • 基础作用
    MySQL标准逻辑备份工具，将库/表导出为SQL文本，支持全实例、单库、单表、仅结构/仅数据备份；InnoDB依靠MVCC实现无锁在线备份。
  • 面试必背核心参数
    参数 作用 面试考点
    --single-transaction InnoDB无锁一致性快照备份 生产环境必备，RR隔离级依靠MVCC，不阻塞业务读写
    --master-data=2 记录备份时binlog偏移位置 用于搭建主从、时间点完整恢复
    --all-databases 导出实例全部数据库 全量备份使用
    -d 仅导出表结构，不含数据 初始化环境脚本
    -t 仅导出数据，无建表语句 增量同步数据
    --routines --triggers 备份存储过程、触发器 容易遗漏的备份项
  • 面试坑点
    --single-transaction仅支持InnoDB引擎；MyISAM不支持快照，会加全局读锁阻塞业务。
