4.1 filter,limit和skip

  • filter 过滤

  • limit 获取前几个元素

  • 注意1：中间方法，返回新的Stream流，原来的Stream流只能使用一次，建议使用链式编程

  • 完整代码
  • 核心知识点说明
    方法名 类型 作用 示例
    filter(Predicate) 中间操作 过滤元素，仅保留符合条件的元素 filter(s -> s.startsWith("张")) 保留张开头的名字
    limit(long n) 中间操作 截取流的前n个元素，截断后续元素 limit(3) 取集合前3个元素
    skip(long n) 中间操作 跳过流的前n个元素，返回剩余元素 skip(4) 跳过前4个，取后面的元素
    forEach(Consumer) 终端操作 遍历流中元素，执行最终操作 forEach(System.out::println) 打印所有元素
  • 关键注意事项
  1. Stream流的一次性：中间操作会返回新的Stream流，原流只能使用一次，因此必须使用链式编程，不能重复复用同一个流。
  2. 不修改原数据：Stream流的所有操作都是对集合数据的副本进行处理，不会影响原集合/数组的内容。
  3. Lambda简化：filter的匿名内部类写法可以用Lambda表达式大幅简化，代码更简洁、可读性更强。
  4. 分页实现：skip(n).limit(m) 是Stream实现分页的标准写法，比如skip(4).limit(3) 对应「第2页，每页3条」的效果。

5.2 collect

  • collect(Collector collector) 收集流中的数据，放到集合中（List Set Map）

  • 完整代码

  • 核心知识点详解
  • 1. collect() 终端操作

  • 作用：将Stream流中的元素，收集到指定的集合（List/Set/Map）中，是Stream流的终端操作，执行后流关闭，不可复用。

  • 核心工具类：java.util.stream.Collectors，提供了toList()、toSet()、toMap()等静态方法，用于生成对应的收集器。

    1. 三种集合收集方式对比
      收集方式 对应方法 集合特性 适用场景
      List Collectors.toList() 可重复、有序 需要保留元素顺序、允许重复的场景
      Set Collectors.toSet() 不可重复、无序 需要自动去重、不关心顺序的场景
      Map Collectors.toMap() 键唯一、键值对结构 需要将元素转为「键-值」映射的场景
    1. toMap() 关键说明（对应第一张图的注释）

  • 参数一（键生成规则）：Function的两个泛型分别为「流元素类型」和「键的目标类型」，apply方法的入参是流中每个元素，返回值作为Map的键。

  • 参数二（值生成规则）：同理，apply方法的返回值作为Map的值。

  • 键冲突处理：Map键必须唯一，若流中存在重复键（如示例的张无忌），必须添加第三个参数(oldVal, newVal) -> newVal，用新值覆盖旧值解决冲突，否则直接抛异常。

  • 运行结果
  • 注意事项
  1. Map键重复问题：直接使用Collectors.toMap(keyMapper, valueMapper)时，若流中存在重复键，会直接抛出IllegalStateException，必须添加冲突处理参数。
  2. split索引越界：确保元素格式严格符合"姓名-性别-年龄"，否则split("-")数组长度不足，会抛出ArrayIndexOutOfBoundsException。

  4. 流的一次性：collect是终端操作，同一个流只能执行一次，不能重复复用。

    Lambda表达式 代码简洁，链式编程，灵活度高 日常开发、简单转换逻辑
    方法引用 代码极简，直接引用类的成员方法/构造方法，可读性强 转换逻辑完全匹配方法、追求极致简洁

  • 三、关键注意事项

  1. 方法引用的前提条件
    • 必须有对应的函数式接口（map依赖Function，toArray依赖IntFunction）
    • 被引用的方法必须存在（Student类必须有getName()方法）
    • 方法的参数/返回值必须和抽象方法完全一致
  2. 数组类型匹配要求
    • toArray(String[]::new) 中的数组类型，必须和流中元素类型完全一致（流中是String，数组必须是String[]）
  3. Stream流的一次性
    • map是中间操作，toArray是终端操作，流执行后会关闭，不可重复使用
  4. Student类的规范要求
    • 必须提供getName()方法，否则Student::getName无法引用
    • 构造方法、Getter/Setter是JavaBean规范，必须完整实现

9. 文件拷贝案例改写

  • 一、完整代码
  • 二、核心知识点解析

  1. 缓冲区大小设计
    byte[] bytes = new byte[1024 * 1024 * 5];

    • 这里定义了一个 5MB 的字节数组作为缓冲区，是批量读写的关键优化。
    • 相比单字节读写，批量读写大幅减少了磁盘IO次数，拷贝效率提升非常明显，尤其适合大文件。

  2. read(bytes) 与 write(bytes, 0, len) 的配合

    • fis.read(bytes)：尝试读取最多 bytes.length 个字节到数组中，返回本次实际读取的字节数 len。
    • fos.write(bytes, 0, len)：只把数组中前 len 个字节写入文件，避免了数组残留旧数据的问题。如果直接写 fos.write(bytes)，当最后一次读取不足数组长度时，会把数组中之前的旧数据也写进去，导致文件损坏。

  3. 资源释放顺序

    • 输入流 fis 和输出流 fos 都需要调用 close() 释放资源，顺序是先创建的后关闭（这里先创建 fis，再创建 fos，所以先关 fos，再关 fis）。

15. 字符输入流原理解析

  • 创建字符输入流对象
    底层：关联文件，并创建缓冲区（长度为8192的字节数组）
  • 读取数据
    底层逻辑：
  1. 判断缓冲区中是否有数据可以读取
  2. 缓冲区没有数据：就从文件中获取数据，装到缓冲区中，每次尽可能装满缓冲区
    • 如果文件中也没有数据了，返回-1
  3. 缓冲区有数据：就从缓冲区中读取。
    • 空参的read方法：一次读取一个字节，遇到中文一次读多个字节，把字节解码并转成十进制返回
    • 有参的read方法：把读取字节、解码、强转三步合并了，强转之后的字符放到数组中
  • 关键逻辑说明
  1. 核心问题：fr.read() 第一次调用会触发缓冲区加载，将文件数据预读到缓冲区（默认8192字节）；后续用FileWriter清空文件后，fr 仍能读取缓冲区中已加载的数据，但文件中原本未读入缓冲区的剩余数据无法再读取（因为文件已被清空，且流已关联旧的文件状态）。
  2. 注意事项：
    • 代码中 FileWriter 构造方法默认会清空文件，这是关键操作。
    • 流的关闭顺序建议：先关写流fw，再关读流fr，避免资源泄漏。

15. 字符输出流原理解析

  • 一、方法定义与核心特性
    成员方法 官方作用 关键特性
    public void flush() 将缓冲区中的数据，刷新到本地文件中 刷新后流对象仍处于可用状态，可以继续调用write()写入数据
    public void close() 释放资源、关闭流通道 执行逻辑：先自动调用flush()刷新缓冲区数据 → 再关闭流通道、释放底层资源。关闭后流对象不可再使用，无法继续写入数据
  • 二、核心区别与使用场景
  1. 功能差异
    • flush() 只负责强制把缓冲区数据写入文件，不关闭流，是“中途同步”操作。
    • close() 会先完成数据刷新，再彻底关闭流、释放文件句柄等资源，是“收尾操作”。
  2. 使用场景
    • 当需要多次写入且中途让数据实时落盘（比如日志实时写入），可以调用flush()避免数据滞留缓冲区。
    • 当所有写入操作完成后，必须调用close()释放资源，防止文件句柄泄漏。
  3. 补充注意点
    • close() 内部已经包含了flush()的逻辑，写入完成后直接调用close()即可，无需重复调用flush()。
    • 调用close()后，流对象无法再执行任何读写操作，否则会抛出IOException。
