15. 字符输出流原理解析

  • 一、方法定义与核心特性
    成员方法 官方作用 关键特性
    public void flush() 将缓冲区中的数据，刷新到本地文件中 刷新后流对象仍处于可用状态，可以继续调用write()写入数据
    public void close() 释放资源、关闭流通道 执行逻辑：先自动调用flush()刷新缓冲区数据 → 再关闭流通道、释放底层资源。关闭后流对象不可再使用，无法继续写入数据
  • 二、核心区别与使用场景
  1. 功能差异
    • flush() 只负责强制把缓冲区数据写入文件，不关闭流，是“中途同步”操作。
    • close() 会先完成数据刷新，再彻底关闭流、释放文件句柄等资源，是“收尾操作”。
  2. 使用场景
    • 当需要多次写入且中途让数据实时落盘（比如日志实时写入），可以调用flush()避免数据滞留缓冲区。
    • 当所有写入操作完成后，必须调用close()释放资源，防止文件句柄泄漏。
  3. 补充注意点
    • close() 内部已经包含了flush()的逻辑，写入完成后直接调用close()即可，无需重复调用flush()。
    • 调用close()后，流对象无法再执行任何读写操作，否则会抛出IOException。

16.最大并行数

  • 一、核心定义
    最大并行数 = 同一时间内，真正同时在CPU上执行的线程数量。
  • 不是「等待的线程」「排队的线程」，只有正在运行的才算并行；
  • 它受两个因素限制：硬件（CPU核心数） + 软件（线程池配置）。
  • 二、分两层理解
    1. 物理最大并行数（硬件上限）
      由你的CPU逻辑核心数决定（这是绝对上限，无法突破）：
  • 比如电脑是 4核8线程（英特尔超线程技术），物理最大并行数 = 8；
  • 单核CPU：物理最大并行数 = 1（只能交替执行线程，没有真正并行）。

一句话：CPU有多少个逻辑核心，就能真正同时跑多少个线程。

    1. Java 代码中的最大并行数（软件控制）
      就是在线程池里配置的：
      ThreadPoolExecutor 的第二个参数 maximumPoolSize（最大线程数）

这是应用层自己设定的、最多能同时运行的线程数量，超过这个数的任务只能排队/拒绝。

  • 三、关键区分：并行 vs 并发（必懂）
    你学多线程一定会混淆，这里一次性分清：
    概念 含义 例子
    并行（Parallel） 同一时刻，真正同时执行多个线程 8核CPU同时跑8个线程
    并发（Concurrent） 同一时间段，多个线程交替执行 单核CPU快速切换2个线程
  • 四、最终实际并行数
    实际并行数 = 最小值(CPU逻辑核心数, 线程池最大并行数)
  • 例：CPU 4核，线程池最大并行数6 → 实际最多并行4个线程
  • 例：CPU 8核，线程池最大并行数3 → 实际最多并行3个线程
  • 五、总结（极简版）
  1. 物理最大并行数：CPU逻辑核心数，硬件天花板；
  2. Java最大并行数：线程池 maximumPoolSize，软件设定的上限；
  3. 我们写代码时，线程池的最大线程数，就是业务层面的最大并行数。

  • 1. Java提供了什么API帮我们创建代理？
    Java原生提供了JDK动态代理API，核心位于java.lang.reflect包下：
  • 核心类：Proxy（专门用于生成代理对象的工具类）
  • 核心方法：Proxy.newProxyInstance()（生成代理对象的静态方法）
  • 配套接口：InvocationHandler（定义代理对象的方法执行逻辑）

这套API只能为实现了接口的类生成代理，也是你之前明星案例中使用的方式。

  • 2. newProxyInstance方法的参数及含义
    （注：你写的newIProxyInstance是笔误，正确为newProxyInstance）
    该方法需要接收3个参数，每个参数的含义如下：
参数序号 参数类型 作用说明 案例对应写法
1 ClassLoader loader 类加载器，用于加载生成的代理类。通常传入被代理对象的类加载器（或当前类的类加载器），保证代理类能被正确加载 ProxyUtil.class.getClassLoader()
2 Class<?>[] interfaces 接口数组，指定代理类需要实现的接口。代理对象会实现这些接口的所有方法，因此必须和被代理对象实现的接口一致 new Class[]{Star.class}
3 InvocationHandler h 调用处理器，代理对象的所有方法调用都会交给该处理器的invoke方法处理，是代理逻辑的核心 new InvocationHandler() { ... }
  • 3. InvocationHandler.invoke方法的调用者与参数

  • 调用者
    invoke方法由生成的代理对象自动调用，你不需要手动调用它：
    当你调用代理对象的任何接口方法时（比如proxy.sing()），代理对象都会自动把这次调用转发给invoke方法，由它来执行增强逻辑+真实方法调用。

  • 参数含义
    invoke方法接收3个参数，每个参数的作用如下：

  1. Object proxy：生成的代理对象本身（就是你调用的那个代理对象，一般很少直接使用）
  2. Method method：当前被调用的接口方法的Method对象（比如调用sing()时，这里就是sing方法的反射对象）
  3. Object[] args：调用方法时传入的参数数组（比如调用sing("只因你太美")时，这里就是["只因你太美"]）
